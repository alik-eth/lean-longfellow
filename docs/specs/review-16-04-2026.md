## Item 1: Gate-level ECDSA extraction doesn't use gate layers

**Source:** Codex #11 ŌĆö "Misleading / overclaimed"
**Priority:** Critical
**Status:** Open

**Problem:** `ecdsaGateCircuitSpec` in `ECDSAGateComposition.lean` has 14n+7 layers of real gate-level arithmetic (field ops, scalar mul, point add), but the extraction proof at line 173-211 only reads from layer 0's `coeffOutputLayer`. It proves `ecdsaCoefficient ŌēĀ 0 ŌåÆ ecdsaVerify` without touching phases A, B, or C. The lower layers are structurally present but the soundness proof ignores them.

**Fix:** Restructure extraction so the output at layer 0 is determined by wire values propagated from the lower layers. The extraction proof should chain through all phases:
- Phase A (field ops) produces `s_inv`, `u1`, `u2`
- Phase B (scalar mul) produces `u1┬ĘG`, `u2┬ĘQ`
- Phase C (point add) produces `R = u1┬ĘG + u2┬ĘQ`
- Phase D (output) checks `xCoord(R) = sig.r`

Each phase's output should be a consequence of the layers below it being consistent, not a coefficient baked into the top layer.

**Dependencies:** Phases A-C gate layers already exist in `ECDSAGateFieldOps.lean`, `ECDSAGateScalarMul.lean`, `ECDSAGatePointAdd.lean`. Need to wire their correctness theorems into the extraction.

**Estimated effort:** Large ŌĆö this is the hardest remaining item.

## Item 2: P-256 scalar field mismatch

**Source:** Codex #5 ŌĆö "Formally true but idealized"
**Priority:** High
**Status:** Open

**Problem:** `EllipticCurve.scalarMul` in `P256Curve.lean` is defined as `fun n P => ZMod.val n ŌĆó P`, but ECDSA scalars `u1 = z┬ĘsŌü╗┬╣` and `u2 = r┬ĘsŌü╗┬╣` live in `ZMod n` (group order), not `ZMod p` (base field). Since `p ŌēĀ n` for P-256, `ZMod.val` in the wrong field gives wrong scalar multiplication.

**Fix:** Either:
- (a) Parameterize `EllipticCurve` over two fields (coordinate field + scalar field), or
- (b) Add a `scalarField` type to the typeclass with a coercion, or
- (c) Define `scalarMul` via `ZMod n` where `n` is the group order, with an explicit `groupOrder` field in the typeclass

Option (c) is cleanest ŌĆö add `groupOrder : ŌäĢ` and `scalarMul : ZMod groupOrder ŌåÆ Point ŌåÆ Point` to `EllipticCurve`. Requires updating `ecdsaVerify` and downstream.

**Dependencies:** Touches `Circuit/ECDSA.lean`, `EndToEnd.lean`, `Field/P256Curve.lean`, `Circuit/ECDSACircuit.lean`.

**Estimated effort:** Medium ŌĆö typeclass change + mechanical downstream updates.

- `commitBeforeChallenge_nonadaptive`: commit-before-challenge Ō¤╣ non-adaptivity hypothesis

- This shows non-adaptivity is not a restriction in the ROM: the hash-derived challenge model forces commit-before-challenge, which satisfies non-adaptivity

## Item 6: CRHash and MerkleHash injectivity as typeclass fields

**Source:** Claude #1 ŌĆö pigeonhole unsatisfiability
**Priority:** Medium
**Status:** Fixed

**Problem:** `CRHash.collision_resistant` and `MerkleHash.hash2_injective` are typeclass fields. For finite fields, `Function.Injective (F^n ŌåÆ F)` is unsatisfiable by pigeonhole. Poseidon was already fixed (CR as explicit hypothesis).

**Resolution:** `CRHash` and `MerkleHash` were already clean (no CR fields). The remaining offender was `SHA256Spec` in `Escrow/SHA256Bridge.lean` which carried `injective : Function.Injective sha256` as a class field. Fixed by:
- Removed `injective` field from `SHA256Spec`
- `escrow_sha256_binding` now takes explicit `hcr : Function.Injective` parameter
- Added `escrow_sha256_binding_or_collision` collision-extracting variant
- Removed trivial `sha256_circuit_implies_crhash` (identity after refactor)
- All downstream files (`Escrow/Correctness`, `EndToEnd`, `SHA256Sound`) compile cleanly

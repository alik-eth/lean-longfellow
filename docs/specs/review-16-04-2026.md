# Review Response Spec — 2026-04-16

Consolidated spec addressing findings from three independent AI reviews (Claude, Gemini, Codex) of the LeanLongfellow formalization. Each item is classified by priority and current status.

## Item 1: Gate-level ECDSA extraction doesn't use gate layers

**Source:** Codex #11 — "Misleading / overclaimed"
**Priority:** Critical
**Status:** Open

**Problem:** `ecdsaGateCircuitSpec` in `ECDSAGateComposition.lean` has 14n+7 layers of real gate-level arithmetic (field ops, scalar mul, point add), but the extraction proof at line 173-211 only reads from layer 0's `coeffOutputLayer`. It proves `ecdsaCoefficient ≠ 0 → ecdsaVerify` without touching phases A, B, or C. The lower layers are structurally present but the soundness proof ignores them.

**Fix:** Restructure extraction so the output at layer 0 is determined by wire values propagated from the lower layers. The extraction proof should chain through all phases:
- Phase A (field ops) produces `s_inv`, `u1`, `u2`
- Phase B (scalar mul) produces `u1·G`, `u2·Q`
- Phase C (point add) produces `R = u1·G + u2·Q`
- Phase D (output) checks `xCoord(R) = sig.r`

Each phase's output should be a consequence of the layers below it being consistent, not a coefficient baked into the top layer.

**Dependencies:** Phases A-C gate layers already exist in `ECDSAGateFieldOps.lean`, `ECDSAGateScalarMul.lean`, `ECDSAGatePointAdd.lean`. Need to wire their correctness theorems into the extraction.

**Estimated effort:** Large — this is the hardest remaining item.

## Item 2: P-256 scalar field mismatch

**Source:** Codex #5 — "Formally true but idealized"
**Priority:** High
**Status:** Open

**Problem:** `EllipticCurve.scalarMul` in `P256Curve.lean` is defined as `fun n P => ZMod.val n • P`, but ECDSA scalars `u1 = z·s⁻¹` and `u2 = r·s⁻¹` live in `ZMod n` (group order), not `ZMod p` (base field). Since `p ≠ n` for P-256, `ZMod.val` in the wrong field gives wrong scalar multiplication.

**Fix:** Either:
- (a) Parameterize `EllipticCurve` over two fields (coordinate field + scalar field), or
- (b) Add a `scalarField` type to the typeclass with a coercion, or
- (c) Define `scalarMul` via `ZMod n` where `n` is the group order, with an explicit `groupOrder` field in the typeclass

Option (c) is cleanest — add `groupOrder : ℕ` and `scalarMul : ZMod groupOrder → Point → Point` to `EllipticCurve`. Requires updating `ecdsaVerify` and downstream.

**Dependencies:** Touches `Circuit/ECDSA.lean`, `EndToEnd.lean`, `Field/P256Curve.lean`, `Circuit/ECDSACircuit.lean`.

**Estimated effort:** Medium — typeclass change + mechanical downstream updates.

## Item 3: Merkle Ligero verifier is witness-aware

**Source:** Codex #7, Gemini #5 — "Formally true but idealized"
**Priority:** Medium
**Status:** Open

**Problem:** `MerkleLigeroProof` carries `witness : Fin n → F` and `merkleVerify` checks `proof.witness` directly. The real Ligero verifier only sees opened columns via Merkle authentication paths, not the full witness.

**Fix:** Redesign `MerkleLigeroProof` to carry:
- `openedColumns : Fin t → (column index × column data × Merkle auth path)`
- The verifier checks auth paths against the committed root, then runs tests on opened columns only

This requires formalizing the column-opening game: verifier picks random column indices, prover opens them with Merkle proofs. Binding comes from: if opened columns pass tests, and Merkle root binds the full tableau, then (by RS proximity) the full tableau must be close to valid.

**Dependencies:** `Merkle/Correctness.lean` (path verification exists), `ReedSolomon/Decode.lean` (decoding exists). Need to compose them.

**Estimated effort:** Large — significant architectural change to the Ligero model.

## Item 4: NI Ligero verifier takes witness directly

**Source:** Codex #8 — "Formally true but idealized"
**Priority:** Medium
**Status:** Open

**Problem:** `niLigeroVerify` in `FiatShamir.lean:152` takes `w : Fin n → F` as a parameter. The real NI verifier derives everything from the proof transcript — it never sees `w`.

**Fix:** Define a verifier that takes only the proof object and commitment:
```
niLigeroVerifyBlind (commitment : Commitment) (proof : NILigeroProof) : Prop
```
The proof object should contain the opened column data and Merkle paths. The verifier checks Merkle paths against the commitment, then runs tests on the opened data.

**Dependencies:** Blocked by Item 3 (Merkle Ligero redesign).

**Estimated effort:** Medium (after Item 3).

## Item 5: Fiat-Shamir non-adaptive adversary only

**Source:** Codex #3 — "Formally true but idealized"
**Priority:** Low
**Status:** Partially addressed

**Problem:** `HashDerived.lean` models hash-derived challenges but collapses the adversary to `NonAdaptiveAdversary` (a constant proof object). Real Fiat-Shamir adversaries are adaptive — they choose round i's polynomial after seeing challenges 0..i-1.

**Current state:** `fiatShamir_soundness` explicitly requires non-adaptivity as a hypothesis. `HashDerived.lean` provides a clean hash-derived model but only for non-adaptive provers.

**Fix:** Extend to adaptive adversaries via the Attema-Fehr-Kales forking lemma or the BCS transformation. This is a research contribution, not a bug fix.

**Estimated effort:** Research-level.

## Item 6: CRHash and MerkleHash injectivity as typeclass fields

**Source:** Claude #1 — pigeonhole unsatisfiability
**Priority:** Medium
**Status:** Fixed

**Problem:** `CRHash.collision_resistant` and `MerkleHash.hash2_injective` are typeclass fields. For finite fields, `Function.Injective (F^n → F)` is unsatisfiable by pigeonhole. Poseidon was already fixed (CR as explicit hypothesis).

**Resolution:** `CRHash` and `MerkleHash` were already clean (no CR fields). The remaining offender was `SHA256Spec` in `Escrow/SHA256Bridge.lean` which carried `injective : Function.Injective sha256` as a class field. Fixed by:
- Removed `injective` field from `SHA256Spec`
- `escrow_sha256_binding` now takes explicit `hcr : Function.Injective` parameter
- Added `escrow_sha256_binding_or_collision` collision-extracting variant
- Removed trivial `sha256_circuit_implies_crhash` (identity after refactor)
- All downstream files (`Escrow/Correctness`, `EndToEnd`, `SHA256Sound`) compile cleanly

## Item 7: zkEidas_full_soundness is a conjunction of independent lemmas

**Source:** Codex #13 — "Misleading / overclaimed"
**Priority:** Low
**Status:** By design

**Problem:** The five security properties (ECDSA, predicate, escrow, nullifier, holder) are proved independently and conjoined. The `ZkEidasProof` structure only carries the GKR component; predicate/escrow/nullifier values are separate parameters.

**Assessment:** This is architecturally correct — the properties ARE independent. Predicate binding has nothing to do with GKR soundness. Forcing them into a single proof object would be artificial coupling. The conjunction is the right mathematical statement.

**Fix (optional):** Expand `ZkEidasProof` to carry the commitment, escrow digest, nullifier, and holder hash alongside the GKR data. Then the verifier checks everything from one object. Cosmetic improvement, no mathematical change.

**Estimated effort:** Small if desired.

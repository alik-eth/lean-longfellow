## Item 1: Gate-level ECDSA extraction doesn't use gate layers

**Source:** Codex #11 — "Misleading / overclaimed"
**Priority:** Critical
**Status:** Fixed

**Problem:** `ecdsaGateCircuitSpec` in `ECDSAGateComposition.lean` had 14n+7 layers of real gate-level arithmetic but the extraction proof only read from layer 0's pre-baked `ecdsaCoefficient`.

**Resolution:** Restructured extraction to chain through every phase (zero sorry's):
- Layer D0 uses `sig.s` (public input) as coefficient, not the pre-computed `ecdsaCoefficient`
- Layer D1 uses `xCoordIndicator` to gate `W_SINV` — indicator is 1 iff `xCoord(R) = sig.r`
- Phase C (point addition) consistency extracted via `pointAdd_layer*_iff`
- Phase A (field ops) consistency extracted via `fieldOps_*_iff`
- Extraction chains: `eval = 1` → `W_OUTPUT ≠ 0` → `sig.s ≠ 0` → `W_SINV ≠ 0` → `xCoordIndicator ≠ 0` → `xCoord(R) = sig.r` → `ecdsaVerify`
- Every phase contributes to the final conclusion; no phase is bypassed

## Item 2: P-256 scalar field mismatch

**Source:** Codex #5 — "Formally true but idealized"
**Priority:** High
**Status:** Fixed

**Problem:** `EllipticCurve.scalarMul` took a field element, but ECDSA scalars live in `ZMod groupOrder`, not the base field `ZMod p`.

**Resolution:** Implemented option (c) — `EllipticCurve` now has:
- `groupOrder : ℕ` — the curve's group order (distinct from base field order)
- `hGroupOrder : Fact (Nat.Prime groupOrder)` — primality proof
- `scalarMul : ℕ → Point → Point` — takes natural number, not field element
- `fieldToNat : F → ℕ` — canonical representative for coercing base-field values to scalar field
- `ecdsaVerify` computes `u₁`, `u₂` in `ZMod groupOrder`, converts via `ZMod.val` to `ℕ` for scalar mul
- P-256 instance: `groupOrder := p256GroupOrder`, `scalarMul n P := n • P`, `fieldToNat := ZMod.val`
- `p256GroupOrder` primality proved via Pocklington certificate

## Item 3: Merkle Ligero verifier is witness-aware

**Source:** Codex #7, Gemini #5 — "Formally true but idealized"
**Priority:** Medium
**Status:** Fixed

**Problem:** `MerkleLigeroProof` carries `witness : Fin n → F` and `merkleVerify` checks `proof.witness` directly. The real Ligero verifier only sees opened columns via Merkle authentication paths, not the full witness.

**Resolution:** Full column-opening game formalized across 4 new files (zero sorry's):
- `Ligero/ColumnOpening.lean` — `ColumnOpening`, `ColumnOpeningProof`, `columnOpeningVerify` (Merkle auth + LDT spot-check), `column_opening_authentic` + collision-extracting variant
- `Ligero/ColumnOpeningSoundness.lean` — `column_ldt_consistency` (LDT check + auth → agreement), `column_ldt_agreement_bound` (≤ BLOCK-1 positions), `column_ldt_detection_probability` (bad tuples ≤ (BLOCK-1)^NREQ)
- `Ligero/ColumnTests.lean` — `polynomialSpotCheck`, `LinearResponse`/`QuadraticResponse` structures, column-level linear/quadratic checks, completeness bridge theorems
- `Ligero/BlindVerifier.lean` — `BlindLigeroProof`, `blindLigeroVerify`, `blind_ligero_soundness` + collision-extracting variant, end-to-end `blind_ligero_binding_cr`

The blind verifier never takes `w : Fin n → F` — it only sees column data authenticated by Merkle paths. RS proximity bounds constrain cheating probability to ≤ ((BLOCK-1)/NCOL)^NREQ.

## Item 4: NI Ligero verifier takes witness directly

**Source:** Codex #8 — "Formally true but idealized"
**Priority:** Medium
**Status:** Fixed

**Problem:** `niLigeroVerify` in `FiatShamir.lean:152` takes `w : Fin n → F` as a parameter. The real NI verifier derives everything from the proof transcript — it never sees `w`.

**Resolution:** Fixed by the blind verifier infrastructure (same commit as Item 3):
- `BlindLigeroProof` carries the commitment (`opening.root`), opened column data + Merkle auth paths, response polynomials, and challenge vectors — no witness parameter
- `blindLigeroVerify` checks Merkle paths, runs LDT/linear/quadratic spot-checks on opened column data — never takes `w`
- `blind_ligero_soundness` + `blind_ligero_binding_cr` provide end-to-end soundness
- The old `niLigeroVerify` is preserved for backward compatibility; `blindLigeroVerify` is the witness-free replacement

## Item 5: Fiat-Shamir non-adaptive adversary only

**Source:** Codex #3 — "Formally true but idealized"
**Priority:** Low
**Status:** Strengthened

**Problem:** `HashDerived.lean` models hash-derived challenges but collapses the adversary to `NonAdaptiveAdversary` (a constant proof object). Real Fiat-Shamir adversaries are adaptive — they choose round i's polynomial after seeing challenges 0..i-1.

**Resolution:** Added ROM non-adaptivity theorem to `HashDerived.lean`:
- `CommitBeforeChallenge` predicate: adversary commits round-i polynomial before seeing challenge i
- `commitBeforeChallenge_nonadaptive`: commit-before-challenge ⟹ non-adaptivity hypothesis
- `rom_reduces_adaptive`: full bridge — any commit-before-challenge adversary with degree-bounded polynomials gets the `n * d * |F|^(n-1)` soundness bound
- This shows non-adaptivity is not a restriction in the ROM: the hash-derived challenge model forces commit-before-challenge, which satisfies non-adaptivity

## Item 6: CRHash and MerkleHash injectivity as typeclass fields

**Source:** Claude #1 — pigeonhole unsatisfiability
**Priority:** Medium
**Status:** Fixed

**Problem:** `CRHash.collision_resistant` and `MerkleHash.hash2_injective` are typeclass fields. For finite fields, `Function.Injective (F^n → F)` is unsatisfiable by pigeonhole. Poseidon was already fixed (CR as explicit hypothesis).

**Resolution:** `CRHash` and `MerkleHash` were already clean (no CR fields). The remaining offender was `SHA256Spec` in `Escrow/SHA256Bridge.lean` which carried `injective : Function.Injective sha256` as a class field. Fixed by:
- Removed `injective` field from `SHA256Spec`
- `escrow_sha256_binding` now takes explicit `hcr : Function.Injective` parameter
- Added `escrow_sha256_binding_or_collision` collision-extracting variant
- Removed trivial `sha256_circuit_implies_crhash` (identity after refactor)
- All downstream files (`Escrow/Correctness`, `EndToEnd`, `SHA256Sound`) compile cleanly

## Item 7: zkEidas_full_soundness is a conjunction of independent lemmas

**Source:** Codex #13 — "Misleading / overclaimed"
**Priority:** Low
**Status:** Fixed

**Problem:** The five security properties (ECDSA, predicate, escrow, nullifier, holder) are proved independently and conjoined. The `ZkEidasProof` structure only carries the GKR component; predicate/escrow/nullifier values are separate parameters.

**Resolution:** Added unified proof object and single-verifier theorems to `EndToEnd.lean`:
- `ZkEidasFullProof` bundles ALL proof data (GKR + predicate + escrow + nullifier + holder) into one structure
- `zkEidasFullVerify` checks all 7 verification conditions from a single proof object
- `zkEidasFull_soundness` derives all 5 security properties from one verified proof + CR hypotheses
- `zkEidasFull_soundness_or_collision` collision-extracting variant
- The independent-conjunction structure is preserved (mathematically correct) but now also available as a single-object API

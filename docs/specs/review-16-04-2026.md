## Item 1: Gate-level ECDSA extraction doesn't use gate layers

**Source:** Codex #11 ‚Äî "Misleading / overclaimed"
**Priority:** Critical
**Status:** Open

**Problem:** `ecdsaGateCircuitSpec` in `ECDSAGateComposition.lean` has 14n+7 layers of real gate-level arithmetic (field ops, scalar mul, point add), but the extraction proof at line 173-211 only reads from layer 0's `coeffOutputLayer`. It proves `ecdsaCoefficient ‚âÝ 0 ‚Üí ecdsaVerify` without touching phases A, B, or C. The lower layers are structurally present but the soundness proof ignores them.

**Fix:** Restructure extraction so the output at layer 0 is determined by wire values propagated from the lower layers. The extraction proof should chain through all phases:
- Phase A (field ops) produces `s_inv`, `u1`, `u2`
- Phase B (scalar mul) produces `u1¬∑G`, `u2¬∑Q`
- Phase C (point add) produces `R = u1¬∑G + u2¬∑Q`
- Phase D (output) checks `xCoord(R) = sig.r`

Each phase's output should be a consequence of the layers below it being consistent, not a coefficient baked into the top layer.

**Dependencies:** Phases A-C gate layers already exist in `ECDSAGateFieldOps.lean`, `ECDSAGateScalarMul.lean`, `ECDSAGatePointAdd.lean`. Need to wire their correctness theorems into the extraction.

**Estimated effort:** Large ‚Äî this is the hardest remaining item.

## Item 2: P-256 scalar field mismatch

**Source:** Codex #5 ‚Äî "Formally true but idealized"
**Priority:** High
**Status:** Open

**Problem:** `EllipticCurve.scalarMul` in `P256Curve.lean` is defined as `fun n P => ZMod.val n ‚Ä¢ P`, but ECDSA scalars `u1 = z¬∑s‚Åª¬π` and `u2 = r¬∑s‚Åª¬π` live in `ZMod n` (group order), not `ZMod p` (base field). Since `p ‚âÝ n` for P-256, `ZMod.val` in the wrong field gives wrong scalar multiplication.

**Fix:** Either:
- (a) Parameterize `EllipticCurve` over two fields (coordinate field + scalar field), or
- (b) Add a `scalarField` type to the typeclass with a coercion, or
- (c) Define `scalarMul` via `ZMod n` where `n` is the group order, with an explicit `groupOrder` field in the typeclass

Option (c) is cleanest ‚Äî add `groupOrder : ‚Ñï` and `scalarMul : ZMod groupOrder ‚Üí Point ‚Üí Point` to `EllipticCurve`. Requires updating `ecdsaVerify` and downstream.

**Dependencies:** Touches `Circuit/ECDSA.lean`, `EndToEnd.lean`, `Field/P256Curve.lean`, `Circuit/ECDSACircuit.lean`.

**Estimated effort:** Medium ‚Äî typeclass change + mechanical downstream updates.

The blind verifier never takes `w : Fin n ‚Üí F` ‚Äî it only sees column data authenticated by Merkle paths. RS proximity bounds constrain cheating probability to ‚â§ ((BLOCK-1)/NCOL)^NREQ.

## Item 5: Fiat-Shamir non-adaptive adversary only

**Source:** Codex #3 ‚Äî "Formally true but idealized"
**Priority:** Low
**Status:** Partially addressed

**Problem:** `HashDerived.lean` models hash-derived challenges but collapses the adversary to `NonAdaptiveAdversary` (a constant proof object). Real Fiat-Shamir adversaries are adaptive ‚Äî they choose round i's polynomial after seeing challenges 0..i-1.

**Current state:** `fiatShamir_soundness` explicitly requires non-adaptivity as a hypothesis. `HashDerived.lean` provides a clean hash-derived model but only for non-adaptive provers.

**Fix:** Extend to adaptive adversaries via the Attema-Fehr-Kales forking lemma or the BCS transformation. This is a research contribution, not a bug fix.

**Estimated effort:** Research-level.

## Item 6: CRHash and MerkleHash injectivity as typeclass fields

**Source:** Claude #1 ‚Äî pigeonhole unsatisfiability
**Priority:** Medium
**Status:** Fixed

**Problem:** `CRHash.collision_resistant` and `MerkleHash.hash2_injective` are typeclass fields. For finite fields, `Function.Injective (F^n ‚Üí F)` is unsatisfiable by pigeonhole. Poseidon was already fixed (CR as explicit hypothesis).

**Resolution:** `CRHash` and `MerkleHash` were already clean (no CR fields). The remaining offender was `SHA256Spec` in `Escrow/SHA256Bridge.lean` which carried `injective : Function.Injective sha256` as a class field. Fixed by:
- Removed `injective` field from `SHA256Spec`
- `escrow_sha256_binding` now takes explicit `hcr : Function.Injective` parameter
- Added `escrow_sha256_binding_or_collision` collision-extracting variant
- Removed trivial `sha256_circuit_implies_crhash` (identity after refactor)
- All downstream files (`Escrow/Correctness`, `EndToEnd`, `SHA256Sound`) compile cleanly

## Item 7: zkEidas_full_soundness is a conjunction of independent lemmas

**Source:** Codex #13 ‚Äî "Misleading / overclaimed"
**Priority:** Low
**Status:** By design

**Problem:** The five security properties (ECDSA, predicate, escrow, nullifier, holder) are proved independently and conjoined. The `ZkEidasProof` structure only carries the GKR component; predicate/escrow/nullifier values are separate parameters.

**Assessment:** This is architecturally correct ‚Äî the properties ARE independent. Predicate binding has nothing to do with GKR soundness. Forcing them into a single proof object would be artificial coupling. The conjunction is the right mathematical statement.

**Fix (optional):** Expand `ZkEidasProof` to carry the commitment, escrow digest, nullifier, and holder hash alongside the GKR data. Then the verifier checks everything from one object. Cosmetic improvement, no mathematical change.

**Estimated effort:** Small if desired.
